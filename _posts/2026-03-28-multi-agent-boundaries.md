最近在想一个问题：什么情况下才真正需要多 Agent？

我的结论是，必须满足以下三条之一，否则都是过度设计。

## 三个合理的拆分时机

**1. 当前模型不具备某种能力**

聊天模型因为成本限制，专业任务需要更强的领域模型才能完成。这时候引入专门的 Agent 是有意义的——不是架构需要，是能力缺口。

**2. 任务之间上下文关系不大**

多 Agent 本质上是一种上下文管理策略。我们之所以拆，是因为不知道怎么清理上下文——多 Agent 更适合"白名单式上下文共享"，只传必要的少量信息。如果两个任务差异足够大，天然适合隔离。

**3. 高吞吐高并行（分布式思路）**

Coordinator-Worker 模式是在传统分布式架构上加了一个智能 Master。以前的 Master 是规则代码写死的，现在换成了推理能力——但分布式的本质没变：高吞吐、高并行，代价是缺乏控制，不适合强耦合任务。

## 单 Agent 先榨干

无论何时，单 Agent 一定要榨干到极致，再考虑多 Agent。

评价标准很简单：在 tool、prompt、skill、context memory 充分调优的基础上看任务完成率。真正的拆分信号不是"完成率低"，而是**"互相掣肘"**——每次改进一个维度都在破坏另一个维度，调优空间的自由度耗尽了。这是架构边界，不是模型天花板，两者要区分清楚。

## 一个有趣的工程类比

Multi-Agent 的演化路径和传统工程完全一致：

```
tool → skill → 子 Agent
函数 → SDK  → 服务拆分
```

有趣的地方在于：当年服务化之后，上下文交换代价暴增——序列化、网络传输、接口契约。现在 Agent 拆分之后，同样面临 context 传递的成本爆炸，我们在用 prompt 压缩、context 蒸馏解决同一个问题，只是换了介质。

但有一个地方 Agent 比服务更难：服务接口是人定义的，可以强制稳定；Agent 之间的"接口"是自然语言或半结构化输出，天然有漂移，随模型版本变化。微服务有 contract testing，Multi-Agent 这套工具链还没成熟。

某种程度上，现在的 Multi-Agent 工程就像 **2005 年的微服务**——思路是对的，但工具链还没跟上，大家还在土法手工管。

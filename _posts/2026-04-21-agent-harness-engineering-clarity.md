最近在做 Agent Harness 的过程中，有几个体会越来越清晰。

## Harness 是一面镜子

做 Agent 工程，本质上是在逼你把每个环节拆解清楚：哪里需要模型决策，哪里用工具，工具调用不好是不是说明描述不够清楚，是不是没用对合适的模型，是不是上下文不足，是不是需要 Skill，是不是需要 Subagent——同时还要考虑成本。

像 Claude Opus 会把简单任务下发给 Sonnet 和 Haiku 这件事，看起来是"模型调度"，实质上是在做任务分解和 Spec 撰写。能让小模型跑起来的前提，是你把任务描述清楚了。这本身就是智能工作中最贵的部分。

## 没有成本压力做不好 Agent

很多团队所谓"等待模型变强"其实是在偷懒。成本压力不只是省钱，它是在强迫你回答一个根本问题：这里到底需不需要智能？很多地方你以为需要模型判断，结果换成规则或工具调用反而更稳定、更快、更便宜。

如果竞争对手能用更便宜的模型做到同等效果，你一样没有竞争力。等模型变强，是在赌别人不会更聪明地用弱模型——这个赌注的赢面越来越小。

## 做 Agent 和训练模型，是同一个问题的两条路

大家都说"模型在吞噬 Agent"，但被吞噬的是什么？是你搞清楚了"这类任务应该怎么拆"这件认知本身。模型从你的 Harness 里学到的，是你的 problem decomposition 方式，不是你的代码。

做 Agent 和训练模型，往往是解决同一个问题的不同途径，需要的领域知识几乎一样。Agent 工程里每一次成功的任务执行，都是一条潜在的训练样本。做 Agent 和做数据飞轮，未来可能会变成同一件事。

现在也许没有人为这些工作直接付费，但它在**提前定义未来模型的能力边界**——总比坐等模型变强有意义。

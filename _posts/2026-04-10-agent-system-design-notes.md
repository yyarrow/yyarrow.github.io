**一个可用的 Agent 系统，本质上只需要四件事：LLM call loop、工具、记忆、验证机制。** 工具做对了，loop 跑起来，记忆不乱，验证兜底——这就是全部起点。以下是最近密集做 Agent 系统积累的实操心得。

---

## 工具设计

**谨慎使用 optional 参数。** 要么必填，要么给合理默认值，不要留灰色地带。每一个 optional 参数背后都是一个需要向模型解释的决策点，解释和测试成本极高。逼自己想清楚：这个参数你希望模型来决策，还是希望调用方指定？

**什么时候拆 tool，什么时候用多参数？** 判断标准是两个 action 的上下文重合度。如果两个 action 的调用场景、前置条件、返回结构差异很大，就拆成两个 tool；否则会引入大量 optional 参数，让 schema 变得混乱。

**tool 暴露什么灵活度，取决于用户能手动做什么。** 用户手动能完成的操作，tool 最好都支持。一旦用户发现"我手动能做到，工具居然做不到"，信任就断了。暴露灵活度不是迁就懒，是防止认知摩擦。

**tool 的命名也是认知成本。** 命名模糊会让模型在选择时产生歧义，本质上是设计者没想清楚这个 tool 的边界。

---

## 标配能力

复杂 Agent 有几个能力是百分之百需要的，无脑整合：**todo、scratchpad（记事板）、skill loader**。而且尽量让用户可见——用户看到 agent 在记什么、在加载什么，才会建立真正的信任感。

---

## 复杂度卸载原则

系统复杂度的归宿有一条链：

> system prompt 太复杂 → 卸载到 skill  
> skill 太复杂 → 抽象成 tool  
> tool 太复杂 → 拆成更细更明确的 tool  
> tool 太多 → 做 tool loader 按需加载

每一层都是在把复杂度结构化，而不是藏起来。tool loader 这一层常被忽略，但 context window 有限，按需加载既省 token，又能降低模型选错 tool 的概率。

---

## 工具的归属

想清楚 tool 是跑在客户端还是服务端，想清楚 memory 和 session 状态存在哪里。**优先级：跟用户走，其次跟 tool 走。** 很多数据混乱和 bug，根源就是没想清楚归属，存错地方了。

---

## 记忆设计

Memory 被过度神话了。拆开看三层：

- **聊天历史**：最基础，但有 token 上限，超了就要截断或摘要
- **Session memory（scratchpad/todo）**：本次任务的工作台，大多数场景到这里就够了
- **Long-term memory**：只有真正需要跨 session 持久化的东西才值得做

todo 和 scratchpad 本质上也是 session memory，只是结构化了一层。

mem0 这类框架的问题是把"存"做得很顺滑，但"找"和"用"的成本被藏起来了——每次都要多花 token 去检索可能根本用不上的内容，召回质量不稳定，还可能把旧的错误记忆带进来干扰当前任务。

真正决定要做 long-term memory 之前，先想清楚三个问题：怎么存、怎么找、愿意在这些"可能用不上的东西"上花多少 token。本质上是一个 ROI 计算：

> **这段记忆被用上的概率 × 它带来的价值 > 存储+检索的 token 成本 + 干扰风险**

大多数场景，这个等式不成立。

---

## 验证与迭代

工具设计、验证机制、loop——这些是 Agent 系统所需要的全部起点。

工具稳定之后，才是评测和回归迭代的时机。工具不稳定时做评测，你分不清是模型的问题还是工具的问题，噪音太大，迭代方向容易跑偏。

变量收敛了（工具行为是确定的），失败才能被归因——是 prompt/reasoning 的问题，还是工具本身的 bug。这时候的回归才有意义，模型能力是变量，工具稳定是底线。
